# AUN P0/P1 最小改动修复方案

日期：2026-04-18
来源：基于 `AUN-SDK与服务模块深度审查结果-2026-04-18.md`、`AUN修复路线图-2026-04-18.md`、`AUN主链二轮细化审查-message-gateway-auth-ca-2026-04-18.md` 以及第三轮主链深挖结果整理。
目标：在**尽量不改对外协议、不大拆架构**的前提下，先把最危险的时序、原子性、幂等问题收敛住，让系统从“线上易炸”降到“失败可解释、重试可控、状态可收敛”。

## 一、总原则

本批修复不追求“架构最优”，先追求三件事：

1. **状态先收敛，再对外宣告成功**
   - 先落 authoritative state，再发 event / push / success response。

2. **副作用操作必须可重试**
   - send、issue_cert、renew、rekey、queue enqueue 这类操作，必须能承受“服务端已处理、客户端以为失败后重试”的场景。

3. **优先最小顺序调整，不先做大重构**
   - 能通过顺序重排、事务包裹、唯一约束、幂等键解决的，不要先引入复杂新组件。

4. **不把未知伪装成成功**
   - 连接断开、下游响应丢失、event publish 失败，这些都应表现成“结果未知/部分成功”，不能继续对外宣称“已送达/已完成”。

## 二、逐条最小改动修复建议

### 1. `auth.aid_login1` 的 pending 注册早于 CA 校验
- 证据：`extensions/services/auth/entry.py:580-604`
- 问题本质：当前是“先进入待验证状态，再去做 authoritative cert 校验”。这会让 pending 生命周期早于服务端信任判断完成。

#### 最小改动目标
让 pending 的正式注册**晚于** `ca.get_cert(aid)` 与公钥/证书归属校验完成。

#### 建议顺序
1. 接收 `aid`、`cert_pem`、`client_nonce`。
2. 先 `await ca.get_cert(aid)` 获取 authoritative cert 记录。
3. 在 auth 侧完成 `aid -> cert -> public key` 一致性校验。
4. 只有校验通过后，再调用 verifier 注册 pending，并生成 `request_id/server_nonce`。
5. `aid_login2` 仅接受“已注册且未消费”的 pending。

#### 事务化要求
- 不需要数据库事务。
- 但 verifier 内部的“注册 pending / 消费 pending / 删除 pending”应在同一个临界区内完成，避免双消费与并发覆盖。

#### 幂等保护
- `request_id` 必须单次消费。
- 同一连接上同 `client_nonce` 的重复 `aid_login1`，要么覆盖旧 pending，要么明确拒绝，不能并存两个有效挑战。
- `aid_login2` 对已消费 `request_id` 必须稳定拒绝。

#### 必补测试
- AID 不存在时不得留下 pending。
- DB cert 与客户端 cert 公钥不匹配时不得留下 pending。
- `ca.get_cert` 超时/失败时不得留下 pending。
- 并发两次 `aid_login1` 不得生成两个可用 pending。
- `aid_login2` 重放必须失败。

### 2. `ca.issue_cert` 并发下可能产生双 `active_signing`
- 证据：`extensions/services/ca/entry.py:449-476`
- 问题本质：典型 check-then-act。两个并发请求都看到“当前不存在 active_signing”，随后各自插入。

#### 最小改动目标
同一 `(aid, curve, key_purpose=signing)` 任意时刻最多只能存在一张 authoritative `active_signing`。

#### 建议顺序
1. 进入数据库事务。
2. 对目标 `(aid, curve, signing)` 记录加锁，或依赖唯一约束作为最终仲裁。
3. 在事务内重新检查当前 active cert。
4. 在同一事务内分配 `version`。
5. 插入新 cert。
6. 提交事务。
7. 若命中唯一约束冲突，返回稳定错误，或读取已存在 active cert 后返回确定性结果。

#### 事务化要求
以下步骤必须放进**同一事务**：
- `get_active_signing_cert`
- `get_next_version`
- `insert_cert`

如果库里已有 `active_signing_slot` 唯一约束，则应用层不要再把“前置查询结果”当成最终真相，真正仲裁交给事务 + 唯一约束。

#### 幂等保护
- 对相同 `(aid, curve, pubkey)` 的短窗口重复签发请求，建议返回稳定结果，而不是继续生成第二张 cert。
- 如果短期内做不到完整幂等返回，至少保证“不会双 active”。

#### 必补测试
- 并发两次 `issue_cert`，只能成功一次。
- 一次成功一次命中唯一约束时，最终库内只有一张 active。
- 重试同一签发请求，不得产生第二张 active。
- 在 version 分配与 insert 之间故障，事务回滚后不留半成品。

### 3. `ca.renew_cert` / `ca.rekey` 的状态切换非原子
- 证据：`extensions/services/ca/entry.py:558-563`、`extensions/services/ca/entry.py:618-629`
- 问题本质：旧 cert 降级、新 cert 插入、`replaces_serial` 更新、旧 cert revoke 是多步分离，任一步失败都可能把 cert 链弄成半成功状态。

#### 最小改动目标
把“旧 cert -> 新 cert”的切换收敛成**单个原子状态迁移**。

#### 建议顺序
1. 进入事务并锁定旧 active cert。
2. 生成新 cert 记录，但先不要让外层看到完成状态。
3. 在同一事务内写入新 cert、更新 `replaces_serial`、更新旧 cert 生命周期状态。
4. 事务提交后，再返回成功。
5. 只有事务提交完成，auth / gateway 才应把新 cert 当作 authoritative active。

#### 事务化要求
以下步骤必须放进**同一事务**：
- 读取旧 active cert
- 分配新 version
- 插入新 cert
- 更新 `replaces_serial`
- 更新旧 cert 生命周期

#### 幂等保护
- renew / rekey 应至少携带一个操作级幂等键，最小可行方案可用：`old_serial + op_type + new_pubkey_digest`。
- 同一个 renew/rekey 请求重复到达时，应返回同一结果，而不是继续发新 cert。

#### 必补测试
- 在新 cert 插入后、旧 cert revoke 前故障，事务必须整体回滚。
- 重试同一 renew/rekey，不得生成多张新 cert。
- 任意故障点后，库内不得同时出现两个逻辑上的 active。
- `replaces_serial` 链必须连续完整。

### 4. `gateway` 断连时直接删 pending，晚到响应会静默丢弃
- 证据：`extensions/services/gateway/ws_server.py:480-485`
- 问题本质：客户端连接一断，gateway 立即把该连接所有 pending 删除；如果 kernel/下游响应稍后才回来，结果就只能被丢弃。

#### 最小改动目标
断连后 pending 不应立即消失，而应进入“连接已丢失、结果待收尾”的终态管理。

#### 建议顺序
1. 连接断开时，不立即 `pop` pending。
2. 先把 pending 标记为 `orphaned/disconnected`。
3. 晚到响应到达时，正常结算该 pending 的最终状态。
4. 如果连接已不存在，则不再尝试回包客户端，但要把结果留到 TTL 清理阶段。
5. 由 `_prune_expired_pending` 统一按 TTL 回收，而不是断连即删。

#### 事务化要求
- 这里主要是**内存状态机原子性**，不是 DB 事务。
- pending 状态至少应有：`active -> disconnected -> resolved/expired`，不要再用“存在/不存在”二态表示。

#### 幂等保护
- 所有有副作用的方法，至少要允许透传 `operation_id` / `message_id`。
- gateway 不能自己吞掉这个幂等键；它必须原样传到下游模块。

#### 必补测试
- 客户端断开后，下游响应晚到，gateway 不得抛异常或静默污染状态。
- 客户端断开后重连并重试同一 `message_id`，服务端不得重复执行副作用。
- pending 过期后应被统一清理，不得永久泄漏。

---

### 5. `message` 跨域 send 先分配本地 `sender_seq` 再 relay
- 证据：`extensions/services/message/entry.py:1257-1300`
- 问题本质：relay 失败时，本地 sender seq 已被消耗；如果 SDK 把这个 seq 当成功语义，就会在失败路径推进本地状态。

#### 最小改动目标
跨域 send 的成功语义必须与 relay 结果一致；失败不能继续消耗“看起来像成功”的 seq 语义。

#### 建议顺序
**推荐最小安全顺序：**
1. 先生成 `message_id`。
2. 先调用 federation relay。
3. 只有 relay 返回接受/成功后，再分配本地 `sender_seq` 并写本地发送记录。
4. 最终返回给 SDK 的 `seq` 只表示“本地 authoritative 已接受”的编号，不表示远端已读或已投递到设备。

如果短期内无法完全重排顺序，则至少要做到：
- relay 失败时返回失败，不返回误导性的成功语义；
- 文档与 SDK 明确：该 `seq` 不是远端交付确认号。

#### 事务化要求
- relay 前后跨服务无法做单库事务。
- 但“本地分配 sender_seq + 本地持久化发送记录”应在单个原子单元内完成。

#### 幂等保护
- `message_id` 必须成为跨域 send 的幂等键。
- 同一 `message_id` 的重试，要么返回既有结果，要么明确返回“处理中/已接受”，不能重复投递。

#### 必补测试
- relay 失败时，接口不得返回 `delivered`。
- 重试同一 `message_id` 不得形成重复跨域发送。
- 断线重试场景下，SDK 不得因为旧 `seq` 误判成功。

### 6. `message._check_aid_exists()` 的 positive cache TTL 过长
- 证据：`extensions/services/message/entry.py:1145-1181`
- 问题本质：`_AID_CACHE_TTL = 600` 让“已撤销/已删除 cert 仍被认为存在”的窗口长达 10 分钟。

#### 最小改动目标
把 cert existence cache 从“长时间可信快照”降为“短时间性能优化”。

#### 建议顺序
1. 把 positive TTL 从 600s 降到 30-60s。
2. negative TTL 再更短，避免不存在状态长时间粘住。
3. CA 超时/异常时不要刷新 positive cache。
4. 对 revoke/delete 敏感场景，优先允许主动失效缓存。

#### 事务化要求
- 无数据库事务要求。
- 核心是缓存更新顺序：只有 authoritative 查询成功时才更新缓存。

#### 幂等保护
- 无额外幂等要求。
- 重点是避免“超时也续命缓存”的 fail-open 变体。

#### 必补测试
- cert 被撤销后，在 TTL 窗口内不得继续长期放行。
- CA 查询失败时，不得刷新正缓存。
- 高并发 send 不得因缓存抖动导致明显性能回退。

### 7. `auth._ws_global_clear()` 直接把所有 pending RPC future 设异常并清空
- 证据：`extensions/services/auth/entry.py:1089-1096`
- 问题本质：连接断开时，auth 直接把所有在途 RPC 置为 `Connection lost`；如果响应其实已在路上，就变成“服务端可能成功，调用方看到失败”。

#### 最小改动目标
把“连接丢失”和“操作一定失败”拆开，不要在断线瞬间把所有在途请求一刀切为失败。

#### 建议顺序
1. 断线时先关闭当前 ws generation，阻止新请求继续走旧连接。
2. 不立即清空 `_pending_rpc`。
3. 允许晚到响应在 TTL 内结算旧请求。
4. TTL 到期后，再把仍未完成的请求置为连接丢失。
5. 对写操作统一依赖幂等键，防止调用方重试造成重复副作用。

#### 事务化要求
- 同样是内存状态机问题。
- 关键是给 pending RPC 增加 generation / terminal state，而不是简单清空 map。

#### 幂等保护
- `auth -> ca` 的写操作必须能承受“响应丢了、客户端重试”。
- 最低要求：同一 renew/rekey/issue 请求可检测重复并稳定返回。

#### 必补测试
- 断线后响应晚到，旧请求不应被二次结算。
- 写操作在“已成功但响应丢失”后重试，不得重复执行。
- 新旧 ws generation 不得串线。

### 8. `mail` 发送队列纯内存，重启直接丢待发邮件
- 证据：`extensions/services/mail/internal/smtp/queue.go:37-87`
- 问题本质：业务层可能已经显示“已入队”，但 worker 队列纯内存，进程一重启，待发邮件直接丢失。

#### 最小改动目标
把“已入队”改成真正 durable 的状态，而不是仅表示“已经塞进内存队列”。

#### 建议顺序
1. 先把 outbound message 与 queue row 同步写入 DB。
2. worker 从 DB 的 pending queue 拉任务，而不是只从内存队列取。
3. 内存队列可以保留，但只能做加速层，不能做 authoritative source。
4. 启动时从 DB 恢复未发送队列。

#### 事务化要求
以下步骤必须在同一事务内完成：
- 写 outbound message 状态
- 写 queue/pending dispatch 记录

#### 幂等保护
- 同一 `message_id` 重复 enqueue，不得产生多条待发记录。
- worker 重试发送时，应通过 message_id / dispatch_id 去重。

#### 必补测试
- enqueue 成功后、worker 尚未发送前进程崩溃，重启后任务必须恢复。
- 同一邮件重复提交不得产生重复待发任务。
- 发送失败重试不得把状态打乱。

## 三、建议批次

### 批次 A：立即止血
1. `auth.aid_login1` 顺序调整
2. `ca.issue_cert` 事务化 + 唯一仲裁
3. `ca.renew/rekey` 原子切换
4. `gateway` pending 改成 `disconnected -> resolved/expired`，不再断连即删
5. `message` 跨域 send 改正 success/seq 语义

### 批次 B：收敛“已处理但响应丢失”
1. `auth._ws_global_clear` 延迟清理 pending
2. `message_id / operation_id` 在 `gateway -> auth/message/ca` 全链透传
3. `message` existence cache TTL 缩短
4. `mail` durable queue 落库

### 批次 C：统一系统语义
1. 统一“成功 / 已接受 / 已送达 / 结果未知”语义词汇
2. 统一所有副作用操作的幂等键规范
3. 补系统级重试测试：断线、超时、晚到响应、重复提交、进程重启

## 四、最值得优先补的测试集

### 1. 模糊结果重试测试
- 请求已被服务端处理，但响应因断线丢失
- 客户端重试同一幂等键
- 结果必须稳定，且副作用不得重复

### 2. 部分失败测试
- DB 成功，但 event publish 失败
- relay 失败，但本地 seq 已分配
- queue enqueue 成功，但 worker 进程崩溃

### 3. 并发测试
- 并发 `issue_cert`
- 并发 `renew/rekey`
- 并发 `aid_login1`
- 并发 send + reconnect + retry

### 4. 重启恢复测试
- `message` dedup / queue / pending 状态在进程重启后的行为
- `mail` 待发队列重启恢复
- CA 状态转换过程中异常退出后的最终 cert 状态

---

## 五、结论

这一批 P0/P1 问题里，真正要先收敛的不是“加更多分支判断”，而是五件事：

1. **先 authoritative 校验，再注册状态**
2. **同一状态迁移必须原子**
3. **断线不能等于副作用失败**
4. **副作用必须有幂等键**
5. **对外成功语义必须降到真实可保证的层级**

只要先把这五件事做对，AUN 主链上最危险的“客户端以为失败而重试、服务端其实已成功”与“客户端以为成功而推进状态、服务端其实没真正交付”两类线上事故，就会显著下降。
