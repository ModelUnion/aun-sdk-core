# AUN message SDK 语义第三轮深度审查

日期：2026-04-18
范围：
- `aun-sdk-core/go`
- `aun-sdk-core/js`
- `aun-sdk-core/ts`
- `aun-sdk-core/python`
- 重点只看与 `message` / `group message` / `gap fill` / `ack` / `reconnect` 相关的实现与语义

目标：不再只看“各语言 SDK 有没有 bug”，而是继续追问：**四种语言 SDK 对 message 主链的理解是不是同一种协议语义**。如果不是，那么即使服务端不改，线上行为也会因为语言差异而分叉。

## 一、第三轮核心结论

四语言 SDK 当前最大的问题，不是每个语言各自都有很多完全独立的大 bug，而是：

1. **它们对 pull 结果、gap fill、ack 时机、解密失败的理解并不完全一致**
2. **有些语言把“服务端返回的原始消息集合”继续往上传，有些语言上传的是“解密后的结果集合”**
3. **有些语言在断线后会停后台循环，有些不会**
4. **有些语言对重试次数、补洞节奏、SeqTracker 恢复顺序定义不完整**
5. **四语言整体上都偏向“解密前 ack”，实际语义更接近 at-most-once，而不是 at-least-once**

这意味着：

- 同一协议，在 Go/JS/TS/Python 客户端上表现不完全一样
- 上层业务如果跨语言开发，很容易遇到“同样功能在一种 SDK 正常，在另一种 SDK 语义不同”的问题

---

## 二、第三轮重点看了什么

### 1. 单聊 pull 语义
- pull 返回的原始 `messages`
- 本地去重 / 排序 / gap 检测
- `onPullResult` 的入参到底是原始消息，还是处理后的消息

### 2. 群聊 pull 与 group E2EE
- `group.pull` 后是否先解密再上抛
- epoch / gap / replay / undecryptable 的事件分叉是否一致

### 3. gap fill 语义
- `_fillP2pGap` / `_fillGroupGap` 或等价路径
- gap 检测后是否重复触发回调
- push 与 pull 命中同一消息时是否会双投递

### 4. ack 时机
- 是在解密前 ack、解密后 ack，还是交付上层后 ack
- 解密失败、签名失败、重放检测失败时是否仍然推进 ack

### 5. 重连与后台任务
- 心跳、拉取循环、补洞循环在断线后是否明确停止
- reconnect 后是否可能出现双 background task 并存

### 6. Retry / max_attempts / SeqTracker
- 是否有限制重试次数
- reconnect 时 seq restore 顺序是否统一
- 多设备与 group cursor 的本地状态恢复是否一致

## 三、跨语言主要差异

## 1. JS：`group.pull` 用解密后的 messages 喂给 `onPullResult`
- 结论：JS 的 `group.pull` 更偏向“把 SDK 已处理后的结果”传给上层，而不是协议层原始返回。
- 问题：
  - 这会把“传输层结果”和“业务层处理结果”混成一个概念。
  - 一旦发生部分解密失败、部分 replay reject，上层拿到的集合与服务端实际返回集合就不再一致。
- 风险：
  - 不同语言 SDK 的 `onPullResult` 语义会分叉。
  - 上层做统计、日志、调试时，JS 看到的是后处理结果，其他语言可能看到原始结果。

### 建议语义
- `onPullResult` 应优先统一为“协议层 pull 结果”。
- 解密后的派生结果应走单独事件或单独字段，而不是直接替换原始 messages。

---

## 2. JS：`_fillP2pGap` / `_fillGroupGap` 有重复调用 `onPullResult` 的风险
- 结论：JS 在 gap fill 路径上，对同一轮补洞返回结果的回调触发更激进，可能形成重复上抛。
- 问题：
  - 正常 pull 已经触发一次
  - gap fill 再补一次时，如果没有做强去重或语义隔离，就可能重复通知上层
- 风险：
  - 上层看见重复 pull result
  - 如果业务层对 `onPullResult` 做副作用处理，会被重复触发

### 性质判断
- 这不一定意味着同一 message 一定被双交付，但至少意味着**回调语义不够干净**。

---

## 3. JS / Python：断线后未显式停止后台任务
- 结论：JS 与 Python 都存在一个倾向：断线时没有把 background heartbeat / pull / retry task 收得足够干净。
- 问题：
  - reconnect 之后可能再起一套新任务
  - 旧任务如果未完全停掉，就会形成双循环并存
- 风险：
  - 双心跳
  - 双 pull
  - 重复补洞
  - 同一消息被多个背景路径处理

### 这类问题的典型表现
- 平时不一定报错
- 但一旦网络抖动、频繁重连，就会慢慢放大成重复回调、重复拉取、状态错乱

---

## 4. Python：缺少明确的 `max_attempts`
- 结论：Python 的某些重试/补洞路径缺乏清晰的最大尝试次数约束。
- 问题：
  - 如果底层条件一直不满足，可能持续重试
- 风险：
  - 背景资源占用持续增长
  - 特定失败条件下进入“长期噪声型重试”

### 影响
- 这不一定马上造成功能错误，但会在弱网、服务端局部异常时制造额外负载和噪声。

---

## 5. Go / JS / TS / Python 整体都偏向“解密前 ack”
- 结论：四语言 SDK 虽然实现细节不同，但总体风格接近：
  - 收到消息或 pull 到消息后，较早推进 ack / cursor
  - 解密失败、事件分叉往往发生在 ack 之后或与 ack 解耦不强
- 这意味着整体语义更接近：
  - **at-most-once**：先推进服务端不再重复给你，再尝试本地消费
  - 而不是 **at-least-once**：本地真正成功处理后再 ack

### 风险
- 一旦本地解密失败、业务层处理失败、UI 层异常，消息可能已经被 ack 掉
- 后续不会自然再收到同一条消息

### 这不是纯 bug，而是协议语义选择
问题在于：
- 当前没有被明确写清楚
- 四语言也没有围绕这个选择做完全一致的事件语义设计

---

## 6. 解密失败路径的事件语义仍不完全统一
- 有的语言会继续把密文或部分消息对象往上传
- 有的语言会发 undecryptable / error 事件
- 有的路径会跳过或弱化错误

### 风险
- 业务层无法写出真正跨语言一致的处理逻辑
- 同样一条“不可解密”消息，在不同 SDK 上可能呈现成不同事件类型

---

## 7. SeqTracker / reconnect 恢复顺序仍未完全统一
- 虽然近期已有修复把部分 SeqTracker 恢复前置，但四语言总体上仍未完全收敛成同一恢复时序：
  1. 恢复本地 seq / keystore / group state
  2. 建连
  3. 认证
  4. 恢复订阅
  5. 补洞
- 风险：
  - 某些语言可能在本地状态尚未完全恢复前，就开始接收正式消息
  - 导致 gap 判定、去重、解密上下文和真正 authoritative 状态错位

---

## 四、按语言看第三轮结论

### Go SDK
- 优点：对时序问题暴露较充分，很多逻辑显式。
- 风险点：
  - push 与 fill gap 并发时仍要非常依赖去重状态正确写入
  - 解密失败事件语义需继续统一
- 总体判断：
  - 机制相对完整，但容易在高并发路径上暴露“去重先后顺序”问题。

### JS SDK
- 风险最集中的点：
  - `group.pull` 的 `onPullResult` 语义偏后处理结果
  - `_fillP2pGap` / `_fillGroupGap` 的回调可能重复
  - 断线后后台任务停得不够彻底
- 总体判断：
  - 功能层面丰富，但“回调语义”和“后台任务生命周期”最需要收敛。

### TS SDK
- 风险点：
  - 更偏底层 transport/connect 时序
  - message 语义相对中性，但仍受统一语义缺失影响
- 总体判断：
  - TS 更像协议骨架，很多问题是“还没被明文化”的系统性问题，而不全是 TS 独有 bug。

### Python SDK
- 风险点：
  - gap fill 与 payload 存在历史耦合问题
  - 后台任务停机与 retry 上限控制不够清晰
- 总体判断：
  - Python 的问题更像“边界条件下容易长期漂移”，不一定每次都炸，但会慢慢积累出不一致。

---

## 五、最关键的统一语义缺口

### 1. `onPullResult` 到底是什么
当前最需要统一的一条就是：

- 它是**协议层原始结果**？
- 还是**SDK 处理后的可交付结果**？

如果四语言不统一，上层永远无法写出一致逻辑。

### 2. ack 代表“收到”还是“成功处理”
当前更像前者，但文义和开发者直觉往往容易把它理解成后者。

### 3. 解密失败是否仍推进 ack
如果是，就应明确承认这是 at-most-once 设计；
如果不是，就必须重构很多补洞/重放逻辑。

### 4. 重连后旧后台任务的生死边界
每一种后台循环都应明确：
- 何时启动
- 何时停止
- 谁拥有它
- reconnect 后是否允许继承

## 六、建议优先级

### 第一优先：统一语义文义
1. 明确 `send` / `pull` / `onPullResult` / `ack` / `undecryptable` 的统一定义
2. 写成跨语言一致约定
3. 再逐语言对齐实现

### 第二优先：先修 JS / Python 背景任务生命周期
1. 断线显式 stop background tasks
2. reconnect 前确保旧任务完全退出
3. 防止双 pull / 双 heartbeat / 双 gap fill

### 第三优先：统一 `onPullResult`
1. 不再把“解密后 messages”直接冒充原始 pull result
2. gap fill 路径避免重复回调
3. 区分 transport result / processed result / delivered event

### 第四优先：统一 ack 语义
1. 明确当前是 at-most-once 还是要转向更强保证
2. 如果维持当前模式，就必须把“解密失败仍可能已 ack”写进协议和 SDK 行为说明

### 第五优先：补恢复顺序与重试上限
1. Python 增加明确 `max_attempts`
2. 四语言统一 SeqTracker / reconnect 恢复顺序
3. 重连时不要边恢复边消费正式消息

---

## 七、建议补的测试

### A. 跨语言一致性测试
- 同一组服务端返回，Go/JS/TS/Python 对 `onPullResult` 的输入应一致
- 同一条不可解密消息，四语言应输出同类事件语义

### B. 断线重连生命周期测试
- 断线 -> reconnect -> 再断线
- 验证旧 heartbeat / pull / gap fill 任务全部退出

### C. gap fill 回调测试
- 正常 pull + gap fill 同时命中同一消息
- 验证上层回调不重复触发

### D. ack 语义测试
- 解密失败后是否仍推进 ack
- 四语言行为必须一致，并与协议文档一致

### E. SeqTracker 恢复顺序测试
- 本地 seq 未恢复完成前，不得开始正式消费消息
- 四语言对同一重连场景结果一致

---

## 八、结论

`message` 相关 SDK 第三轮最重要的发现，不是某一个语言单独“坏掉了”，而是：

1. **四语言 SDK 还没有真正共享一套完全一致的 message 语义**
2. **JS 在 `onPullResult` 与 gap fill 回调语义上最容易和其它语言分叉**
3. **JS / Python 在断线后后台任务生命周期上最容易积累长尾问题**
4. **Python 的 retry 上限约束不够清晰**
5. **四语言整体都更接近“解密前 ack”的 at-most-once 语义，但这一点没有被系统性写清楚**

如果只挑最该先统一的三件事，就是：

- `onPullResult` 的定义
- 断线后后台任务的停止机制
- ack 与解密失败的关系

这三件事如果不先统一，那么即使服务端主链继续修，跨语言 SDK 的行为仍然会持续分叉，最终变成“协议看起来一致，但不同语言客户端体验不一致”的长期维护问题。
