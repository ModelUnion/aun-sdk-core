### P1 — 尽快修复（业务语义 / 状态一致性）

#### P1-1. group `ack_messages` 重复定义，旧版本被覆盖

- 位置：`extensions/services/group/service.py:1651`、`extensions/services/group/service.py:4304`
- 原清单编号：4.18 新审查 #12
- 问题：同名方法定义两次，后者覆盖前者。旧版无 `device_id` 语义成为死代码；历史调用方若仍按旧参数调用，会直接报 `device_id is required`。
- 影响：接口语义漂移，兼容性不可靠，后续维护者容易误判实际生效代码。
- 修复方向：删除旧版或合并两版逻辑，统一到一份明确的 device cursor 语义实现。
- **状态：已修复（文档过期）** — 2026-04-19 验证：仅存在一个定义（`service.py:4297`），测试 `test_ack_messages_only_one_definition` 通过。

#### P1-2. group `ack_messages` 用缓存 `group.message_seq` 做上界截断

- 位置：`extensions/services/group/service.py:1661-1668`
- 原清单编号：H12
- 问题：`seq=min(seq, group.message_seq)` 使用的是 `_require_group` 返回的缓存快照，TTL 30 秒。缓存滞后时会把客户端合法 ack 截断到更小值。
- 影响：客户端已消费消息却无法推进 ack，表现为重复拉取、cursor 停滞。
- 修复方向：改为 DB 实时查询 message_seq，或在 repository 层做 `LEAST(request_seq, current_db_message_seq)`。
- **状态：已修复（文档过期）** — 2026-04-19 验证：当前实现（`service.py:4297-4314`）无 `min(msg_seq, group.message_seq)` 截断逻辑，测试 `test_ack_messages_no_snapshot_truncation` 通过。

#### P1-3. group `add_member` 与 membership_index 双写不原子

- 位置：`extensions/services/group/service.py:644-662`，`extensions/services/group/repository.py:662-729`
- 原清单编号：H11、4.18 新审查 #14
- 问题：repository 层已支持 `index_item` 同事务写入，但 service 层还在先 `add_member()` 再 `_upsert_membership_index()`，两步分离。
- 影响：主成员表与索引表可能分叉；异常中断时只写成功一半。
- 修复方向：service 层接入 `index_item`，把 member + membership_index 一次提交。
- 附带问题：`join_msg_seq/join_event_seq` 仍使用旧 group 快照，建议也改为基于最新 DB 序号。
- **状态：已修复（H11）** — 2026-04-19 验证：`repository.py:662-729` 已实现同事务写入（H11 修复标记），`service.py:643-666` 的 `_add_member_internal()` 已构造 `index_item` 并传入 `repo.add_member()`，测试 `test_add_member_atomic_index_write` 通过。

#### P1-4. group epoch range 边界使用旧快照

- 位置：`extensions/services/group/service.py:1027-1041`
- 原清单编号：4.18 新审查 #13
- 问题：epoch 轮转成功后，`close_epoch_range` / `create_epoch_range` 仍使用函数开头读取的 `group.message_seq` / `group.event_seq`。
- 影响：并发写消息时，epoch 范围可能重叠或出现空洞，进而影响群 E2EE 解密定位。
- 修复方向：CAS 成功后重新读取最新 seq，再写 epoch range。
- **状态：已修复** — 2026-04-19 验证：`service.py:1031-1051` CAS 成功后调用 `await self._require_group(group_id=group_id)` 获取 `fresh_group`，`close_epoch_range` 和 `create_epoch_range` 使用 `fresh_group.message_seq/event_seq`，测试 `test_rotate_e2ee_epoch_uses_fresh_seq_after_cas` 通过。

#### P1-5. auth `token_store._load_latest()` 读取失败会缓存空字典

- 位置：`extensions/services/auth/token_store.py:175-179`
- 原清单编号：M18
- 问题：JSONL 读取异常时返回 `{}` 并写入 1 小时缓存。
- 影响：缓存期内所有合法 token 都会被判定不存在，形成整站拒绝登录。
- 修复方向：读取失败时直接抛异常；不要污染缓存；调用方返回明确的 `token_store_io_failure`。

#### P1-6. auth renew/rekey 的 pending challenge 未先 pop

- 影响：aid/nonce 不匹配时 pending 不会立即失效，可被反复尝试。

#### P1-7. Python SDK 群消息 push 路径带 payload 时发现 gap 不补洞

- 位置：`python/src/aun_core/client.py:1020-1027`
- 原清单编号：4.18 新审查 #13 / 问题 9
- 问题：`if need_pull and not payload_present:` 导致只有不带 payload 的通知才会触发 `_fill_group_gap()`。
- 影响：如果收到带 payload 的 seq=5，而 seq=3/4 缺失，gap 只会被记录，不会主动补拉。
- 修复方向：只要 `need_pull` 为真就应触发补洞；是否已有 payload 不应影响历史 gap 拉取。

#### P1-8. Python SDK replay guard 失败时 fail-open

- 位置：`python/src/aun_core/client.py:1609-1622`
- 原清单编号：M6/M8、4.18 新审查 #14
- 问题：服务端 replay guard RPC 失败时 `return True` 放行。
- 影响：服务端不可达时，跨重启/跨进程重放保护失效。
- 修复方向：改为 fail-close，或至少降级为“本地 guard 命中则拒绝，远端失败时打高优先级错误并阻止分发”。

#### P1-9. message.ack 事件发布失败时 DB 已更新但接口报错

- 位置：`extensions/services/message/entry.py:1902-1934`
- 原清单编号：新审查 #20
- 问题：ack_seq 先更新 DB，再 `_publish_event`。如果事件发布失败，接口报错，但 DB 中 ack 已推进。
- 影响：调用方看到失败后可能重试；发送方却收不到 ack 事件，状态不一致。
- 修复方向：要么承认这是“通知失败但 ack 已成功”的语义并返回部分成功；要么把事件持久化成 outbox 异步补发。

#### P1-10. mail 发送队列纯内存，重启丢待发邮件

- 位置：`extensions/services/mail/internal/smtp/queue.go:37-87`
- 原清单编号：4.18 新审查 #23
- 问题：队列仅保存在进程内 slice 中，`Stop()` 不做持久化。
- 影响：服务重启后排队中的外发邮件全部丢失。
- 修复方向：改为 DB/outbox 持久队列；worker 从 DB 拉待发项并做重试状态机。

#### P1-11. mail 联邦入站仅信任 `_caller_id==gateway`，缺二道校验

- 位置：`extensions/services/mail/internal/core/handler.go:615-713`
- 原清单编号：H17
- 问题：mail 模块未对联邦入站邮件做签名/来源再校验，只依赖上游 gateway。
- 影响：一旦 gateway 被误配或被攻破，mail 无法阻断伪造联邦邮件。
- 修复方向：在 mail 侧增加来源签名校验、域校验或最少做一次可验证的 federation envelope 复验。
