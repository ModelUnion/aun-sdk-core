# AUN 修复路线图

日期：2026-04-18
来源：基于 `aun-sdk-core` 与 `extensions/services` 第一轮深度审查结果整理。
目标：按“先堵最危险洞，再修状态机，再统一语义”的顺序推进修复。

## 一、总体原则

本轮问题的共性不在于“缺功能”，而在于“状态推进顺序错了”。

高频模式包括：

1. **check-then-act**
   - 先检查，后执行，中间没有锁或事务
   - 典型：mail quota、aid_custody AID 占用、ca.issue_cert

2. **双写不原子**
   - 主表与索引、主状态与派生状态不在同一事务
   - 典型：group.add_member + membership_index

3. **先返回成功，后做真正持久化/补偿**
   - 典型：mail 入队、附件落盘、部分 SDK 本地状态与服务端状态脱节

4. **先异步补偿，后写去重标记**
   - 典型：Go SDK pushedSeqs / fillGap

5. **先拿旧快照，后在新状态下使用**
   - 典型：group epoch range 边界、join_msg_seq / join_event_seq

6. **恢复顺序错误**
   - 典型：SeqTracker / transport.connect / reconnect / group E2EE state 恢复顺序

## 二、第一阶段：立即止血

目标：先修会直接造成认证绕过、业务错误、严重数据不一致的问题。

### 2.1 修 `aun_console` 验证码旁路
- 文件：`extensions/services/aun_console/server.py:295-310`
- 原因：
  - 这是直接安全洞，不是普通 bug
- 优先级：**最高**
- 验收：
  - 删除旁路后，所有验证码流量必须走正式校验路径
  - 不再允许手机号后 6 位直接通过

### 2.2 修 `group ack_messages` 重复定义
- 文件：
  - `extensions/services/group/service.py:1651`
  - `extensions/services/group/service.py:4304`
- 原因：
  - 当前类内重复定义，本质属于接口行为不确定
  - 会直接影响所有群 ack 语义
- 修法建议：
  - 统一保留一个入口
  - 明确兼容策略：到底支持 `seq`，还是 `msg_seq + device_id`，还是同时兼容后做归一化
- 验收：
  - 旧 SDK / 新 SDK 的群 ack 行为都有明确测试
  - 不再依赖 Python 方法覆盖顺序

### 2.3 修 `mail quota` 的 TOCTOU
- 文件：`extensions/services/mail/internal/store/quota.go:107-113`
- 原因：
  - 会直接破坏 quota 约束
- 修法建议：
  - quota 检查与 message insert 合并到同一事务
  - 不要 `commit` 后再 insert
- 验收：
  - 并发 APPEND / SMTP 入站测试下，quota 不可超额

### 2.4 修 `mail` 附件错误吞没
- 文件：`extensions/services/mail/cmd/mail-server/main.go:388-403`
- 原因：
  - 会制造“邮件成功但附件丢失”的坏状态
- 修法建议：
  - `storage.put_object` / `InsertAttachment` 任一失败都要上抛
  - 不允许 SMTP 外层在附件失败时返回成功
- 验收：
  - 附件存储失败时，整封邮件状态明确失败或回滚
  - 不存在 silent partial success

### 2.5 修 JS SDK 群 epoch async 判断错误
- 文件：
  - `aun-sdk-core/js/src/client.ts:2086,2088`
  - `aun-sdk-core/js/src/e2ee-group.ts:1122`
- 原因：
  - 这是实锤逻辑错
  - 会引发 epoch rotate 风暴
- 修法建议：
  - 先修 `await`
  - 再补并发 rotate 测试
- 验收：
  - 多 admin 下只有一个兜底 rotate 成功发起
  - 其余节点能正确观测 epoch 已变化

### 2.6 修 Python SDK 群消息 gap 不补
- 文件：`aun-sdk-core/python/src/aun_core/client.py:1029-1033`
- 原因：
  - 可能直接丢历史消息
- 修法建议：
  - `need_pull` 与 `payload_present` 解耦
  - 空洞存在就补，不该看有没有 payload
- 验收：
  - 带 payload 的乱序群消息也能自动补洞

## 三、第二阶段：修核心状态机

目标：修最容易在重连、多设备、并发写入时出错的链路。

### 3.1 修 `group` 的 epoch range 边界快照问题
- 文件：`extensions/services/group/service.py:1027-1041`
- 风险：
  - 会影响群 E2EE 的 epoch 判定
- 修法建议：
  - CAS 成功后，重新读取 authoritative group seq / event seq
  - 不再继续使用旧快照闭合 old/new epoch range
- 验收：
  - epoch rotate 与并发 send 下，消息只会落入唯一正确 epoch

### 3.2 修 `group.add_member` 的双写不原子
- 文件：`extensions/services/group/service.py:644-662`
- 风险：
  - 成员主表、membership index、join boundary 会分叉
- 修法建议：
  - 优先把 membership index 写入并进主事务
  - 至少做到“成员写入 + 索引写入 + join 边界快照”同一原子单元
- 验收：
  - 中途失败不会留下半成功状态

### 3.3 修 SDK 重连恢复顺序
- 重点文件：
  - `aun-sdk-core/ts/src/transport.ts:84-124`
  - `aun-sdk-core/js/src/client.ts:2271,2273,2820-2828`
  - `aun-sdk-core/go/client.go:3228-3231,3256`
  - `aun-sdk-core/python/src/aun_core/client.py` 重连相关路径
- 风险：
  - SeqTracker、pending request、subscription、E2EE state 的恢复顺序混乱
- 修法建议：
  - 统一原则：
    1. 恢复本地 seq / keystore / group state
    2. 建连
    3. 认证
    4. 重建订阅
    5. 再补洞
  - 不要边恢复边开始接收正式消息
- 验收：
  - 单聊 / 群聊 / 重连测试都不重复、不漏、不乱序

### 3.4 修 Go SDK pushedSeqs 去重竞态
- 文件：`aun-sdk-core/go/client.go:1220-1276`
- 风险：
  - 消息重复投递
- 修法建议：
  - 先登记去重标记，再启动 fill gap
  - 或引入更强的单消息处理状态机
- 验收：
  - 推送 + 补洞同时发生时，同一消息只交付一次

### 3.5 修 `leaderboard` 乱序覆盖
- 文件：
  - `extensions/services/leaderboard/repository.py:474-475`
  - `514-515`
  - `542-543`
- 风险：
  - `last_event_at` 回退
- 修法建议：
  - `ON DUPLICATE KEY UPDATE` 时用时间条件保护
  - 仅当新事件更“新”时更新 `last_event_*`
- 验收：
  - 乱序 ingest 不会回退最后活跃时间

### 3.6 修 `aid_custody` 的 AID 占用 TOCTOU
- 文件：
  - `extensions/services/aid_custody/service.py:1234-1237`
  - `extensions/services/aid_custody/repository.py:366-367`
- 风险：
  - 并发抢占 AID 归属
- 修法建议：
  - 数据库唯一约束 + 原子 upsert 语义
  - 不要依赖“先查可用、后写入”
- 验收：
  - 并发导入同一 AID 时，只会有一个成功占有

## 四、第三阶段：统一语义与补测试

目标：让 SDK 和服务端的时序语义收敛，不再各自为政。

### 4.1 统一 group cursor / ack / retention 语义
- 重点字段：
  - `cursor.current_seq`
  - `server_ack_seq`
  - `join_seq`
  - retention floor
- 问题：
  - 当前各语言 SDK 对这些字段的理解并不完全一致
- 结果要求：
  - 统一协议语义
  - 各语言 SDK 一致实现

### 4.2 统一“解密失败”的事件语义
- 问题：
  - Go / JS / TS / Python 有的继续投密文，有的静默跳过，有的发 undecryptable
- 要求：
  - 定义唯一标准：
    - 正常消息
    - 不可解密消息
    - 签名失败消息
    - 重放拒绝消息
  - 不能继续混用

### 4.3 补三类关键测试

#### A. 时序测试
- 重连时 seq restore / connect / subscribe / pull 顺序
- push 与 pull 同时命中同一消息
- epoch rotate 与在途消息交错

#### B. 并发测试
- add_member 并发
- quota 并发
- AID 占用并发
- ack / pull 并发

#### C. 部分失败测试
- 附件写入失败
- CA 超时
- storage 超时
- event publish 失败
- DB commit 成功但后续步骤失败

## 六、结论

最该优先修的，不是“某个功能点少了”，而是这些链路上的状态推进顺序。只要把：
- 原子性
- 幂等性
- 恢复顺序
- 失败语义
- 多设备 cursor 语义

这五件事收敛住，AUN 当前的大多数高危时序 bug 都会显著下降。
