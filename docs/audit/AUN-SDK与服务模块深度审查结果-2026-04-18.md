## 一、真实调用链 DAG

### 1. 认证/证书链
- `SDK -> gateway -> auth -> ca`
- 关键场景：握手 nonce、AID 登录两阶段、token/JWT 验证、证书查询/续期/rekey、CRL 刷新。

### 2. 单聊消息链
- `SDK -> gateway -> message -> ca`
- 跨域时：`message -> gateway.forward_federation`
- 关键场景：send / pull / ack、多设备、重连、seq/cursor、证书存在性检查。

### 3. 群组链
- `SDK -> gateway -> group -> (ca / storage / gateway federation)`
- 关键场景：建群、加成员、退群、群消息、group_events、device cursor、群 E2EE epoch 轮转与密钥分发。

### 4. 存储/附件链
- `SDK -> gateway -> storage`
- Mail 使用：`mail -> storage.put_object/get_object`

### 5. 邮件链
- `SDK/SMTP/IMAP -> mail -> (ca / storage / Mail DB / send queue)`
- 关键场景：SMTP 入站、SMTP 出站、IMAP 会话、附件落盘、quota、投递状态。

### 6. 辅助链
- `nameservice -> auth.verify`
- `leaderboard <- 各类事件`
- `aid_custody -> repository / auth_service`
- `aun_console -> 聚合多个模块 HTTP/RPC`

## 二、第一轮高风险问题总览

> 说明：这里收敛第一轮最值得优先处理的问题，特别偏重时序/状态机问题。

### P0 / P1：服务端

1. **group 的 `ack_messages` 重复定义，旧接口被新接口覆盖**
   - 证据：`extensions/services/group/service.py:1651`、`extensions/services/group/service.py:4304`
   - 影响：旧调用方如果还传 `seq`，会直接走错语义；兼容性与业务行为都不可靠。

2. **group 的 epoch 范围边界使用旧快照，存在 read-after-write 窗口**
   - 证据：`extensions/services/group/service.py:1027-1041`
   - 影响：CAS 轮转成功后若并发有新消息提交，epoch 范围可能记录错误，导致群 E2EE 依据错误 epoch 解密。

3. **group.add_member 与 membership_index 双写不原子**
   - 证据：`extensions/services/group/service.py:644-662`
   - 影响：成员主表和索引表可能分叉；同时 `join_msg_seq/join_event_seq` 使用旧 group 快照，存在加入边界偏差。

4. **group 对 E2EE capability 的判断过度依赖客户端自报**
   - 证据：`extensions/services/group/service.py:639-641`
   - 影响：可把实际能力不足的成员加进 E2EE 群，后续产生密钥分发/解密/状态一致性问题。

5. **mail quota 检查与真正插入分离，存在 TOCTOU**
   - 证据：`extensions/services/mail/internal/store/quota.go:107-113`
   - 影响：并发 APPEND/收信时可能都通过 quota 检查，最终超额写入。

6. **mail SMTP 入站附件落盘失败被静默吞掉**
   - 证据：`extensions/services/mail/cmd/mail-server/main.go:388-403`
   - 影响：SMTP 外层返回成功，但附件可能没写到 storage/DB，形成“邮件成功但附件丢失”。

7. **mail 发送队列纯内存，重启直接丢待发邮件**
   - 证据：`extensions/services/mail/internal/smtp/queue.go:37-87`
   - 影响：业务状态可能已显示“已入队”，但服务崩溃/重启后待发邮件永久丢失。

8. **leaderboard 对 `last_event_id/last_event_at` 无条件覆盖，乱序事件会回退最后活跃时间**
   - 证据：`extensions/services/leaderboard/repository.py:474-475`、`514-515`、`542-543`
   - 影响：乱序到达的旧事件会把“最后活动时间”写回过去。

9. **aid_custody 下载审批状态是纯内存，重启即丢；AID 归属写入还有 TOCTOU**
   - 证据：
     - 内存审批：`extensions/services/aid_custody/service.py:47`、`643-648`
     - 归属抢占：`extensions/services/aid_custody/service.py:1234-1237`、`extensions/services/aid_custody/repository.py:366-367`
   - 影响：审批重启失效；并发下可能发生 AID 归属覆盖。

10. **aun_console 存在验证码旁路**
    - 证据：`extensions/services/aun_console/server.py:295-310`
    - 影响：`code == phone[-6:]` 可直接放行生成 token，属于严重安全问题。

### P0 / P1：SDK

11. **JS SDK 群 epoch 兜底轮转逻辑写错：`currentEpoch()` 是 async，但调用处没 `await`**
    - 证据：`aun-sdk-core/js/src/client.ts:2086,2088`；`aun-sdk-core/js/src/e2ee-group.ts:1122`
    - 影响：非 leader 节点会错误判定 epoch 未变化，触发多 admin 并发 rotate，造成 CAS 风暴和密钥分发混乱。

12. **TS SDK `transport.connect()` 存在 `open/timeout` 竞态**
    - 证据：`aun-sdk-core/ts/src/transport.ts:84-124`
    - 影响：可能出现连接 promise 悬挂、stale ws、重连状态机错乱。

13. **Python SDK 群消息 push 路径：带 payload 的消息发现 gap 也不会补洞**
    - 证据：`aun-sdk-core/python/src/aun_core/client.py:1029-1033`
    - 影响：历史消息空洞可能永久不补。

14. **Python SDK replay guard 失败时 fail-open**
    - 证据：`aun-sdk-core/python/src/aun_core/client.py:1625-1628`
    - 影响：只要 replay guard RPC 失败，就可能放过重放消息。

15. **Go SDK push 路径里，补洞 goroutine 可能先于 `pushedSeqs` 标记，导致重复投递**
    - 证据：`aun-sdk-core/go/client.go:1220-1276`、`1305-1374`
    - 影响：同一条消息可能被推送路径和补洞路径各交付一次。

16. **Go SDK 某些解密失败路径会把密文继续当正常消息投递**
    - 证据：`aun-sdk-core/go/client.go:1250-1264`、`2075-2122`
    - 影响：上层事件语义被污染，本应走 undecryptable 路径的消息被当作正常消息处理。

## 三、时序问题的共性模式

本轮发现大量问题都属于以下模式：

1. **check-then-act**
   - 先检查，后执行，中间没有锁/事务
   - 典型：mail quota、aid_custody AID 占用、ca.issue_cert

2. **双写不原子**
   - 主表更新和索引/派生状态更新不是一个事务
   - 典型：group.add_member + membership_index

3. **先返回成功，后做真正持久化/补偿**
   - 典型：mail 入队、附件落盘、部分 SDK 本地状态与服务端状态脱钩

4. **先异步补偿，后写去重标记**
   - 典型：Go SDK pushedSeqs / fillGap

5. **先拿旧快照，后在新状态下使用**
   - 典型：group epoch range 边界、join_msg_seq/join_event_seq

6. **恢复顺序错误**
   - 典型：SeqTracker / transport.connect / reconnect / group E2EE state 恢复顺序
