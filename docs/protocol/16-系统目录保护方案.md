# 16. 系统目录保护方案

## 16.1 目标

`memberdata` 与 `group_data` 是 Group FS 与 Storage 协作使用的系统级目录名。它们不能被当作普通用户目录处理，否则会造成权限绕过、视图层与真实存储层混淆、误删成员数据和配额统计失真。

本方案定义两个目录名的保护规则：

- `memberdata`：Group FS 视图层的虚拟协议目录。
- `group_data`：Storage 真实层的内部协议根目录。

外部用户、SDK、CLI 和普通 `storage.*` / `storage.fs.*` RPC 只能看到业务语义，不应把真实存储根作为应用目录树入口。`group_data` 的存在不是保密目标；保护目标是目录树不展示系统根、普通写入不能绕过 `group.fs.*` 授权路径。

## 16.2 目录分层

| 名称 | 所在层 | 示例 | 性质 | 对外可见性 |
| --- | --- | --- | --- | --- |
| `memberdata` | Group FS 视图层 | `group_aid:/memberdata/alice.aid/logs/a.md` | 虚拟目录和成员槽位边界 | 在 Group FS 中可见；普通 Storage 面不因目录名隐藏，但写保护 |
| `group_data` | Storage 真实层 | `member_aid:/group_data/{group_aid}/logs/a.md` | 内部真实存储根 | 普通 Storage 目录树隐藏；读/下载不刻意隐藏；写保护 |

`memberdata/{member_ref}` 映射到 `member_aid:/group_data/{group_aid}`。SDK 和 CLI 不得拼接 `group_data`；服务端 group 模块负责视图路径解析，storage 模块负责真实路径保护和配额统计。

## 16.3 总体原则

1. `memberdata` 根和成员槽位不是普通目录，不能被普通文件操作破坏。
2. `memberdata` 不是隐藏目录；保护重点是禁止普通 Storage 写入 group AID 命名空间下的 `memberdata/...`。
3. `group_data` 是保留真实根，普通 Storage 目录树浏览不展示它，普通 Storage 写入、删除、重命名、挂载、授权和状态变更必须拒绝。
4. `group_data` 的对象读取、下载 ticket 和 URL 生成不因路径名被刻意隐藏，但仍必须按普通 read ACL、token、private、owner 规则鉴权。
5. `group_data` 占用的空间必须计入真实 owner AID 的 Storage 配额。
6. 隐藏和保护只改变可见性与入口，不改变数据归属、配额归属和审计归属。
7. 成员数据写入只允许成员本人通过 `group.fs.*` 写自己的槽位子路径。
8. `group.fs.*` 不允许删除 `memberdata` 根、成员槽位根或对应的 `group_data/{group_aid}` 根。

## 16.4 `memberdata` 保护规则

`memberdata` 在 Group FS 视图中表现为虚拟目录和成员槽位边界。普通 Storage 面不因为路径名是 `memberdata` 而隐藏读、stat 或列表结果；但 group AID 命名空间下的 `memberdata/...` 不能被普通 Storage 写入口直接创建、修改、删除或替换。

### 16.4.1 允许行为

| 操作 | 路径 | 结果 |
| --- | --- | --- |
| `stat/lstat` | `group_aid:/memberdata` | 返回虚拟目录节点 |
| `ls` | `group_aid:/memberdata` | 返回成员槽位虚拟节点列表 |
| `stat/lstat` | `group_aid:/memberdata/{member_ref}` | 返回成员槽位节点 |
| `ls/find/df` | `group_aid:/memberdata/{member_ref}` | 通过 group 服务映射到成员真实 `group_data/{group_aid}` |
| `mkdir/cp/mv/rm` | `group_aid:/memberdata/me/path` | 成员本人可操作自己的槽位子路径 |
| `cp` 下载 | `group_aid:/memberdata/{member_ref}/path` | 合法群成员按读权限下载 |

### 16.4.2 拒绝行为

| 操作 | 路径 | 要求 |
| --- | --- | --- |
| `mkdir/rm/mv/cp` | `group_aid:/memberdata` | 必须拒绝 |
| `rm/mv/cp` 覆盖槽位 | `group_aid:/memberdata/{member_ref}` | 必须拒绝，不能删除或替换槽位根 |
| 写他人槽位 | `group_aid:/memberdata/{other}/path` | 必须拒绝 |
| 普通 Storage 写 group 空间 `memberdata/...` | `group_aid:/memberdata/...` | 必须拒绝，禁止绕过 group.fs；普通读/列表不因目录名隐藏 |
| 普通挂载到槽位子路径 | `memberdata/{aid}/subdir` | 必须拒绝，只允许精确槽位根挂载语义 |

`rm group_aid:/memberdata/{member_ref}` 不得解释为删除成员真实数据。若需要解除成员槽位挂载，应使用明确的 `group.fs.umount` 或等价控制面语义。

## 16.5 `group_data` 保护规则

`group_data` 是成员个人 Storage 内的内部协议根，用于承载该成员在某群的成员数据。它不是普通应用目录。

### 16.5.1 普通 Storage 面行为

普通 `storage.*`、`storage.fs.*`、`aun fs` 和 Storage VFS 对 `group_data` 的处理分三类：

- 目录树浏览类入口必须隐藏：`storage.fs.list/find/stat/lstat/resolve/df`、`list_objects`、`list_prefixes`、`list_children`、`get_folder` 等在浏览 owner 目录结构时不得展示 `group_data` 根或其子树；直接指向 `group_data/...` 的树视图请求应返回 not found。
- 读/下载类入口不刻意隐藏：`get_object`、`head_object`、`get_object_url`、`create_download_ticket`、`check_access(read)` 不应仅因路径位于 `group_data/...` 而返回 not found；它们仍按普通 read ACL、token、private、owner 和群成员读代理规则鉴权。
- 写入和状态变更类入口必须拒绝：`put_object`、`append_object`、`set_object_meta`、`delete_object`、`check_upload`、`create_upload_session`、`complete_upload`、`mkdir`、`copy`、`move`、`rename`、`remove`、`delete`、`symlink`、`mount`、`set_acl`、`remove_acl`、`set_visibility`、`issue_token`、`revoke_token` 等作用于 `group_data` 或其子路径时必须拒绝，除非请求来自可信 `group.fs.*` 内部授权路径，或来自 `collab.*` 对 `.collab-versions` / `.collab-snapshots` 等协作内部对象的受控写入。

`df owner:/` 的总量必须包含 `group_data` 占用；`df owner:/group_data/...` 属于目录树 scoped 浏览，普通请求应返回 not found。

上述隐藏和拒绝由服务端 storage RPC 强制实现；CLI、SDK VFS 和其他客户端只呈现服务端返回结果，不要求实现独立的系统目录识别或保护分支。

对外错误建议：目录树隐藏返回 not found；写入和状态变更拒绝返回权限错误。调试模式或内部审计日志可以记录 reserved path 拒绝原因。

### 16.5.2 Group FS 内部访问

可信 `group.fs.*` 内部调用可以访问 `group_data/{group_aid}`：

- `_internal_caller_id == "group.fs"`。
- `_trusted_internal_caller_id == "group"`。
- `_group_fs_group_aid` 必须与 `group_data/{group_aid}` 根一致。
- `_group_fs_storage_owner_aid` 必须是成员 AID。
- `_group_fs_storage_bucket` 与当前 bucket 匹配。
- `_group_fs_storage_path` 必须覆盖当前真实路径 scope。
- 写操作要求 `_group_fs_actor_aid == _group_fs_storage_owner_aid`。
- 读操作要求 actor 是合法群成员，跨域读取必须通过 group 代理上下文校验。

group.fs 内部访问也不能删除 `group_data/{group_aid}` 根本身，只能操作根以下的业务子路径。

`collab.*` 只能在已解析的 `collab_root=group_aid:/memberdata/{member_aid}/...` 范围内写入协作内部对象；协作写授权通过 `collab.set_acl/remove_acl` 面向具体 AID 管理，不允许客户端直接调用 `storage.set_acl/remove_acl` 操作 `group_data`。

## 16.6 配额与计费

`group_data` 内的所有对象、目录元数据和派生 blob 引用必须计入真实 owner AID 的 Storage 配额。

规则如下：

1. `member_aid:/group_data/{group_aid}/...` 的大小计入 `member_aid`，不计入 `group_aid`。
2. group owner、admin 或其他成员通过 group.fs 读取成员数据，不改变配额归属。
3. group.fs 上传到 `memberdata/me/...` 时，写前配额检查使用成员本人的 quota。
4. `df group_aid:/memberdata/{member_ref}` 可以返回成员槽位视图，但其中 quota 字段必须明确表示真实 owner 是 `member_aid`。
5. 普通 `aun fs df member_aid:/` 可以统计 `group_data` 占用，但默认不展示目录名；总量必须包含它。
6. 群解散、成员退出或权限变化不会自动把 `group_data` 从配额中排除。

这保证系统目录保护不会变成绕过个人存储限额的通道。

## 16.7 生命周期与删除策略

| 场景 | 数据处理 |
| --- | --- |
| 成员退出或被踢出群 | 保留其个人 `group_data/{group_aid}` 数据；group.fs 不再允许群内其他成员访问 |
| 群解散 | 默认不删除成员个人 `group_data/{group_aid}` 数据 |
| 成员主动清理 | 必须通过明确的成员数据清理流程，不通过普通 Storage 路径暴露 `group_data` |
| 槽位卸载 | 只移除视图/挂载关系，不删除真实数据 |
| 管理员清理 | 必须有独立授权和审计，不复用普通文件删除语义 |

如果未来提供“清理某群成员数据”能力，应使用语义化 API，例如 `group.fs.purge_member_data` 或用户侧隐私清理入口，并要求二次确认、审计记录和配额变更事件。

## 16.8 CLI / SDK / RPC 要求

### CLI

- `aun fs` 的列表、访问错误和拒绝结果由 `storage.fs.*` 服务端返回决定，不新增专门的 `group_data` 保护逻辑。
- `aun group fs` 的 `memberdata` 视图和根/槽位根破坏性操作拒绝由 `group.fs.*` 服务端返回决定。
- 错误消息按通用错误映射呈现，不鼓励用户拼接真实路径。

### SDK VFS

- Storage VFS 不新增 `group_data` 特判，继续调用普通 `storage.fs.*`。
- GroupFSVFS 只传 `group_aid:/memberdata/{member_ref}/...` 视图路径并调用 `group.fs.*`。
- 四语言 SDK 均不得包含 `memberdata -> group_data` 的客户端映射逻辑，也不得把系统目录保护实现前移到客户端。

### RPC

- 普通 `storage.*` / `storage.fs.*` 对 reserved root fail-closed。
- `group.fs.*` 是唯一可间接访问成员数据的公开 RPC 面。
- storage 内部只信任服务端注入的 group_fs context，不信任客户端提交的同名字段，跨域 relay 时必须保持可信来源校验。

## 16.9 审计与可观测性

Storage 应记录以下内部审计事件：

- 普通目录树入口隐藏 `group_data`。
- 普通读/下载入口访问 `group_data` 按 read 权限鉴权，不因目录名隐藏。
- 普通写入和状态变更入口访问 `group_data` 被拒绝。
- 普通入口写 group 空间 `memberdata` 被拒绝。
- group.fs 代理读取成员槽位。
- group.fs 成员本人写入槽位子路径。
- 尝试删除 `memberdata` 根、槽位根或 `group_data/{group_aid}` 根。

审计记录应包含 requester、actor、group_aid、storage_owner_aid、operation、view_path、storage_path、decision 和 reason。普通用户可见错误不必暴露真实 storage_path。

## 16.10 测试验收

必须覆盖以下测试：

- 普通 storage 写 `group_data/{group_aid}/a.txt` 被拒绝。
- 普通 storage owner 对 `group_data/{group_aid}/a.txt` 的 `get_object/head_object/create_download_ticket/check_access(read)` 按普通读权限成功。
- 普通 storage `mkdir/rm/mv/cp/symlink/mount` 涉及 `group_data` 被拒绝。
- 普通 `ls/find/stat/lstat/resolve/list_children/list_objects/list_prefixes` 不返回 `group_data` 项或对子路径返回 not found，但 `df /` 总量包含其占用。
- 普通 Storage 读/list/stat `memberdata/...` 不因目录名隐藏；普通写 `memberdata/...` 被拒绝。
- `group.fs.cp local -> group_aid:/memberdata/me/a.txt` 成功，配额扣在成员 AID。
- `group.fs.rm group_aid:/memberdata/me/a.txt` 成功。
- `group.fs.rm group_aid:/memberdata/me` 被拒绝或转为非删除真实数据的卸载语义。
- 成员不能写他人 `memberdata/{other}`。
- 非成员不能通过 group.fs 读写任意 `memberdata` 槽位。
- 群解散或成员退出后，普通 Storage 仍不暴露 `group_data`，group.fs 访问按成员关系 fail-closed。

## 16.11 与相关协议的关系

- Group 子协议负责定义 `memberdata` 视图、成员槽位权限和 `group.fs.*` 控制面。
- Storage 子协议负责定义 `group_data` 真实根保护、配额统计、普通 storage 面的目录树隐藏、读/下载鉴权和写保护行为。
- SDK 文档只描述 `memberdata` 视图路径，不暴露真实根作为应用开发入口。
