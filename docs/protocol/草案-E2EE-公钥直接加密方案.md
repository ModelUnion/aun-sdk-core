# 草案 — E2EE 下一代方案（公钥直接加密）

> 状态：设计讨论中
> 日期：2026-05-13
> 相关规范：`08-AUN-E2EE.md`、`08-AUN-E2EE-Group.md`
> 相关议题：`议题-E2EE移动端性能与可用性分析.md`

---

## 1. 背景与目标

现行 E2EE 方案（`08-AUN-E2EE.md` v2.0-draft）以 Prekey + 临时 ECDH 为核心，追求消息级前向安全（PFS）。在移动端群聊场景下暴露出严重的可用性问题（详见 `议题-E2EE移动端性能与可用性分析.md`）：

- 群聊单次发送热路径 5 RPC，epoch 不对齐时退化为 N 次 + 5 s 阻塞
- 群密钥分发硬绑定 P2P E2EE 通道 → 依赖对端**此刻在线**
- 移动端后台即断连，"在线成员"长期接近 0 → 密钥恢复常年超时

本方案的目标：**以 AID 证书公钥为核心的直接加密**，换取可用性，主动放弃消息级 PFS。

---

## 2. 对"前向安全（PFS）"的重新评估

### 2.1 PFS 的理论威胁模型（很窄）

PFS 仅在以下条件**同时**成立时有意义：

1. 攻击者此刻拿不到设备
2. 受害者**本地不留明文**（密文随用随解、立刻擦）
3. 密文在网络 / 服务端被长期存档
4. 长期私钥在"未来某天"才泄露，而不是"当时"

### 2.2 任一条件不成立 → PFS 收益归零的逻辑链

这是一个闭环悖论：

```
追求 PFS  → 要求临时密钥
临时密钥保存 → 一样可被泄露（只是把"一个长期私钥"变成"N 个临时私钥"）
临时密钥不保存 → 自己看不到历史消息
自己要看历史 → 必须本地明文或本地可解密存档
本地存档存在 → 攻击者拿到设备即可读历史 → PFS 保护失效
```

**PFS 保护的"网络密文"，如果本地已解密存档，保护就失去了实际意义。**

### 2.3 Signal 等"PFS 模范"的真实处理

Signal 为了让用户看历史，采用**本地加密 SQLite + 设备钥匙串**存历史消息。结果：

- 设备被攻破 → 钥匙串 → 历史全部可读（PFS 在此攻击下没用）
- 纯网络攻击 + 未来私钥泄露 → PFS 有用

PFS 真实价值只在**国家级对手大规模截流 + 事后定向拿到某目标设备之前历史**这个狭窄威胁模型下。消费级 / Agent 通信级场景不是主要风险。

### 2.4 AUN 现行 Prekey PFS 本就打了折

`08-AUN-E2EE.md` §9.1：

> 接收方 **MUST** 保留旧 prekey 私钥至少 7 天（解密在途消息）

发送侧 ephemeral 用完即丢 —— PFS 成立。
接收侧 prekey 私钥保留 7 天 —— 这 7 天内 PFS 含金量已大打折扣。

换言之现行方案**已经为工程可用性妥协过一轮 PFS**，不是"纯净 PFS"。

### 2.5 AUN 场景下的特殊权衡

| 损失类型 | 触发条件 | 损失量级 |
|---|---|---|
| 一阶损失 | AID 私钥泄露 → Agent 身份被冒充 | **身份灾难**（Agent 作为社会人的身份本体被攻破） |
| 二阶损失 | + 同时满足 §2.1 四条件 → 历史消息额外可解 | 在身份灾难之上的"顺便"损失 |

一阶损失已经是灾难级，二阶损失的边际意义有限。

### 2.6 结论

**在 AUN Agent 通信场景下，将 PFS 从"硬性要求"降为"可选加分项"是合理的工程取舍。**

换回来的是：
- 不依赖 prekey 轮换与补充
- 不依赖对端此刻在线
- 群密钥分发可走服务端离线邮箱（零信任保持不变）
- RPC 次数大幅下降

## 3. 基本设计方向

**核心**：用 AID 证书公钥（P-256）为基础的直接加密，完全抛弃 Prekey / group_secret / epoch / mailbox 这套中间层。每条消息自包含所需解密信息。

### 3.1 基础假设（已澄清）

| 假设 | 含义 |
|---|---|
| 所有设备共用同一个 AID 证书私钥 | 不需要设备级子证书，公钥只有一个 |
| 消息到本地后以明文保存 | Agent 上下文天然保留历史，密文保存无意义，也就无所谓"PFS 保护历史" |
| 证书轮换导致解码失败时懒加载 | 不保留旧私钥，解密失败即触发拉新证书重试 |
| 不做新旧方案共存 / 迁移 | 产品位发布，无包袱 |
| 放弃 group_secret / epoch / mailbox | 每条消息自包含，不需要"群共识密钥" |

### 3.2 P2P 场景流程

**发送方**：拿到对方 AID 证书公钥（本地缓存命中则 0 开销，未命中则一次 HTTPS 拉取 + PKI 全链验证并缓存）。临时生成一对一次性椭圆曲线密钥，用自己的一次性私钥和对方 AID 公钥做一次 ECDH，得出一个秘密值，派生出这条消息专用的对称密钥。用这把对称密钥加密消息正文。用自己的 AID 私钥对密文做一次签名。把"一次性公钥 + 密文 + 签名"打包发出。一次性私钥立即丢弃。

**接收方**：从消息里取出发送方的一次性公钥，用自己的 AID 私钥与它做一次 ECDH，得到同样的秘密值，派生出同一把对称密钥，解密。用发送方 AID 证书公钥验签。

**关键性质**：
- 零交互——发送方单程发出，接收方单程收解，不需要事先握手或对方在线
- 一次性密钥只为"让每条消息对称密钥不同"，不为 PFS；本地不保存
- 冷启动一次 HTTP 拉证书，之后长期缓存

### 3.3 群聊场景流程（Hybrid 多收件人加密）

**发送方**：为这条消息随机生成一把"主对称密钥"，用它把消息正文加密一次（全群共享同一份正文密文）。对群里每个成员重复做一次 P2P 式的小动作——临时生成一对一次性钥匙、用一次性私钥和该成员 AID 公钥做 ECDH、派生出一把"仅对该成员有效的小钥匙"、用小钥匙把**主对称密钥**加密成一个小包。最终组装出：一份正文密文 + 一张"收件人清单"（清单上每一项写着某成员 AID 对应的一次性公钥 + 加密后的主密钥小包）+ 发送方签名。一次 `group.send` 发出。

**接收方**：在收件人清单里找写着自己 AID 的那一项，用自己的 AID 私钥和该项上的一次性公钥做 ECDH，派生出小钥匙，解开小包拿到主对称密钥，再用主对称密钥解开正文密文。用发送方 AID 证书公钥验签。

**关键性质**：
- 消息正文只加密一次，主密钥按成员数 N 次封装（Hybrid Encryption 标准模式）
- 无 epoch、无 group_secret、无邮箱、无 key_request、无任何群共识操作
- 不需要任何成员此刻在线；接收方何时上线就何时独立解密
- 成员变更自然生效：踢人/退群者不再出现在后续清单上；新成员看不到加入前历史；证书轮换后发送方自动用新公钥加密

### 3.4 成员列表与证书同步

- 进群瞬间：一次批量 `group.get_members` + 一次批量 `bulk_get_certs` 拉取全体成员证书，之后本地缓存
- 成员变更：服务端推送 `group.members_changed` 事件，客户端增量拉取新成员证书
- 发送时冲突：服务端在 `group.send` 落地时比对"清单覆盖的 AID" vs "真实成员"，不一致则拒绝并回传新列表，发送方对差集补做小包后重试一次
- 解密失败：若对方证书在本地缺失或 fingerprint 不匹配（证书已轮换），懒加载触发一次拉取重试

### 3.5 发送方认证

保留现行 `sender_signature` 机制：发送方用 AID 私钥对密文和关键字段做 ECDSA-SHA256 签名，接收方通过 `sender_cert_fingerprint` 查本地证书验签。缺失或验签失败一律拒绝消息。

### 3.6 签名覆盖范围（已定：方案 C 折中签）

签名不仅覆盖消息正文密文，还通过一个**清单摘要**覆盖整个收件人清单。这样可以在不与服务端合法元数据编辑冲突的前提下，让接收方能检测服务端对加密层内容的任何篡改。

**签名输入三段拼接**：

1. **密文段**：`ciphertext || tag`（AES-GCM 加密后的消息正文与认证标签）
2. **AAD 段**：规范化序列化后的 AAD 字节（group_id / from / message_id / timestamp / suite 等）
3. **清单摘要段**：对"收件人清单"按规范序列化后的 SHA-256 摘要

**清单的规范化规则**：
- 按成员 AID 字典序升序排列
- 每项包含该成员的 AID、一次性公钥、nonce、wrapped 主密钥密文（整段，不仅是公钥）
- 使用与 AAD 一致的 Canonical JSON（键递归排序 + 紧凑格式 + UTF-8 直出）
- 对最终 JSON 字节取 SHA-256

**消息新增字段**：
- `recipients_digest`：清单摘要（十六进制），供接收方重算比对

**接收方验证流程**：
1. 按规范重算清单摘要，与消息中的 `recipients_digest` 比对，不一致则拒绝
2. 用发送方证书公钥验证 `sender_signature` 是否覆盖 `ciphertext || tag || AAD_bytes || recipients_digest`
3. 确认自己 AID 在清单里
4. 解密自己那一项，再解密消息正文

**wrapped 主密钥必须整段纳入摘要的原因**：若摘要仅包含 `{aid, ephemeral_public_key}`，服务端理论上可篡改某成员的 wrapped_key 密文而不破签，导致该成员解出垃圾但签名看似有效；整段纳入摘要杜绝此路径，成本仅多一次 SHA-256（200 人群 ≈ 0.1 ms）。

**防御范围**：
| 攻击 | 防御 |
|---|---|
| 服务端删除某成员小包 | ✅ 摘要变化，破签 |
| 服务端替换某成员的 wrapped_key | ✅ 摘要变化，破签 |
| 服务端新增收件人（冒名） | ✅ 摘要变化，破签 |
| 服务端篡改正文密文 | ✅ AEAD tag 失败 |
| 服务端附加自己的路由元数据（seq、server_time 等） | ✅ 不影响——这些字段在签名域外 |

### 3.7 清单项指纹与证书版本识别

#### 3.7.1 目的

服务端需要在 `group.send` 落地时比对"发送方清单里每个成员用的证书版本" vs "该成员当前活跃证书版本"，不一致则报 `cert_rotated`，让发送方补发。

#### 3.7.2 指纹存储位置

| 位置 | 存什么 | 由谁维护 |
|---|---|---|
| 成员自己 | 当前证书 PEM + 指纹 | 成员轮换证书时本地更新 |
| 群服务端成员表 | 每个成员的**当前证书指纹**（由成员在证书轮换时上报） | 服务端被动接收，不主动管理证书 |
| 发送方本地缓存 | 每个对端的 `{cert, fingerprint}` | 发送方按需拉取、懒加载刷新 |

**服务端不签发、不审计、不管理证书本身**，只存一个指纹字符串作为"当前版本标记"。这与 AUN"证书可能由其他网关签发"的事实兼容。

#### 3.7.3 指纹格式（截断）

指纹使用 **SHA-256 前 64 位**（16 hex）：

```
sha256:a3b2c1d4e5f60718
```

抗碰撞强度 32 bit 足够本场景（仅作查表 key，攻击者无法构造他人证书的哈希碰撞）。截断比完整指纹节省 55 字节/项。

#### 3.7.4 清单项结构

每个小包增加 `peer_cert_fingerprint` 字段：

```json
{
  "aid":                   "dave.agentid.pub",
  "peer_cert_fingerprint": "sha256:a3b2c1d4e5f60718",
  "ephemeral_public_key":  "base64(...)",
  "nonce":                 "base64(...)",
  "wrapped_key":           "base64(...)"
}
```

单项体积从 ~230 B 增加到 ~276 B（+20%）。

#### 3.7.5 签名域

`peer_cert_fingerprint` **必须纳入 §3.6 的清单摘要计算**。若不入摘要，服务端可篡改某成员的指纹，诱导虚假 `cert_rotated`，让发送方无谓补发（DoS）。入摘要后任何篡改即破签。

#### 3.7.6 服务端处理

- 收到 `group.send`，对清单里每项比对 `peer_cert_fingerprint` vs 成员表里该成员的当前指纹
- 不匹配 → 放入响应 `cert_rotated`，附带服务端知道的 `new_cert_fingerprint`（亦为 64 位截断）
- 匹配 → 正常投递

#### 3.7.7 发送方处理响应

- 收到 `cert_rotated` 列表 → 对每个 AID 用附带的 `new_cert_fingerprint` 做指纹锚点拉取新证书
- 拉到新证书后更新本地缓存，在补发清单中使用新公钥加密

### 3.8 冲突处理与二次补发

#### 3.8.1 核心原则

- **宽松送达**：服务端对清单做"∩ 真实成员"的交集分发，不因不一致而拒绝消息
- **响应带 delta**：每次 `group.send` 响应都捎带成员差集，发送方无需额外调 `group.get_members` 同步
- **懒加载成员列表**：不强制进群时批量同步；首次发送若本地没有成员缓存，发送方可以按"最小清单"（仅含自己）发出，依赖响应补齐（注：此为兜底路径，推荐首次发送前先调 `group.get_members + bulk_get_certs`）
- **二次补发**：对响应中 `cert_rotated` 和 `uncovered_members` 自动补发一次
- **发送方已非成员**：服务端直接拒绝，不走 delta 机制

#### 3.8.2 请求

`group.send` 请求新增字段 `members_version`（客户端本地已知的成员版本号，首次发送或缓存缺失时填 `0`）。

#### 3.8.3 响应

```json
{
  "message_id": "gm-xxx",
  "status": "accepted",
  "request_members_version": 42,
  "server_members_version": 45,
  "members_delta": {
    "added":   ["carol.agentid.pub", "..."],
    "removed": ["bob.agentid.pub",   "..."]
  },
  "delivered_count": 18,
  "dropped_ghosts":  ["bob.agentid.pub"],
  "cert_rotated": [
    {"aid": "dave.agentid.pub", "new_cert_fingerprint": "sha256:..."}
  ],
  "uncovered_members": ["carol.agentid.pub"]
}
```

字段说明：
- `request_members_version` — 发送方本次请求所基于的版本号（回显，便于诊断与幂等）
- `server_members_version` — 服务端此刻的当前版本
- `members_delta` — 从 `request_members_version` 到 `server_members_version` 的变更集合
- `dropped_ghosts` — 清单里有但已非成员的 AID（被服务端静默丢弃，不计入送达；可由 `members_delta.removed ∩ recipients` 推出，冗余保留便于客户端直接使用）
- `cert_rotated` — 清单里有但 fingerprint 已过期的成员，附带新证书指纹，发送方应重拉新证书后补发
- `uncovered_members` — 发消息那一刻已在群、但本次清单没列到的成员，发送方应补发给他们
- `delivered_count` — 实际分发的人数

**关键：`uncovered_members` 以消息发送时刻为时间基准**——服务端扫描当前成员表，仅把 `joined_at ≤ aad.timestamp` 且不在本次清单里的成员放入此列表。`joined_at > aad.timestamp` 的新加入者天然不属于这条消息的目标群体，既不投递也不报告。这保证了补发的收敛性（见 §3.8.6）。

**投递成功的定义**：服务端将该成员的消息副本加入其消息队列即算投递成功，不等待接收方在线或确认。离线成员上线后从队列拉取，不进入补发流程。

若 `request_members_version` 与 `server_members_version` 差距超过服务端 changelog 保留窗口：
```json
"members_delta": {"need_full_sync": true}
```
客户端应调用一次 `group.get_members` 做全量同步。

#### 3.8.4 发送方对响应的处理

1. 应用 `members_delta` 到本地缓存并更新 `members_version`
2. 对 `cert_rotated` 按 `new_cert_fingerprint` 拉取新证书替换本地缓存
3. 对 `uncovered_members` 懒加载补齐本地证书（若缺）
4. 组装补发消息，清单 = `cert_rotated ∪ uncovered_members`（结构见 §3.8.5）
5. 通过同一个 `group.send` 接口发出

#### 3.8.5 二次补发

补发消息 = 原消息的同一条，除了以下两项，其余全部保持一致：

- `recipients`：仅含原响应中的 `cert_rotated ∪ uncovered_members`
- `recipients_digest` 与 `sender_signature`：按新清单重新计算、重新签名

保持不变的项：`message_id`、`ciphertext`、`tag`、`aad`（包含 `from` / `timestamp` / `suite` 等全部字段）、主对称密钥。

补发通过同一个 `group.send` 接口发出。服务端**完全无状态**处理：按清单投递即可，不依赖"原消息是否存在"或"这是否是补发"这类上下文。每个接收方只会收到一次（原消息或补发其中之一），感知不到"这是补发"。

#### 3.8.6 补发的响应与收敛性

补发是普通 `group.send`，响应结构与 §3.8.3 完全同构。

**补发最多一轮，数学上必然收敛**：
- `uncovered_members` 以原消息 `aad.timestamp` 为时间基准过滤，目标群体在发消息那一刻就已固定，不随后续成员加入变化
- 第 1 轮原消息 → 遗漏集 = `cert_rotated ∪ uncovered_members`
- 第 2 轮（补发）→ 新证书已拉、`uncovered` 已覆盖，响应里两项都应为空

因此**协议硬约束：补发仅允许 1 次**（即"原消息 + 1 次补发"共 2 轮）。客户端在第 2 次响应中仍收到非空 `cert_rotated` / `uncovered_members` 时，停止补发并上报业务层——这是极端罕见场景（证书短时间内两次轮换等），由业务层决定是否整体重发。

#### 3.8.7 已退群的发送方

服务端检测到发送方非当前群成员时：
```json
{"error": "not_member", "server_members_version": 45}
```
客户端应停止对该群的发送并通知用户。不进入 delta / 补发流程。

#### 3.8.8 消息时间戳的新鲜度校验

服务端 **MUST** 校验 `aad.timestamp` 在服务端当前时间的 ±5 分钟窗口内，超出则拒绝 `group.send`。防止恶意客户端用伪造时间戳操纵 `uncovered_members` 的过滤基准，也天然挡了重放攻击。

#### 3.8.9 成员版本号

- 单调递增整数，每次成员变更（加入 / 退出 / 踢出 / 证书轮换）+1
- 服务端维护 `changelog` 环形缓冲（建议保留最近 200 次变更，具体由实现定）
- 差距超过缓冲窗口时响应 `need_full_sync: true`

## 4. 性能与体积评估

### 4.1 RPC 次数对比

| 场景 | 现行方案 | 新方案 |
|---|:---:|:---:|
| P2P 热路径 | 1 | **1** |
| P2P 冷启动首条 | 3（get_prekey + 证书 HTTP + send）+ PKI | 1 + 1 HTTP + PKI（首次缓存证书） |
| 群聊热路径 | **5**（get_epoch ×3 + get_members + send） | **1** |
| 群聊 epoch 不对齐 | 6 ~ N + 最多 5 s 阻塞 | 不存在该状态 |
| 群聊密钥恢复 | get_epoch ×N + 多成员 P2P key_request + 5 s 超时 | 不存在该路径 |
| 新加入群 | 逐步恢复（多次 get_epoch + key_request） | 1× get_members + 1× bulk_get_certs |
| 依赖对端在线 | 是 | **否** |

### 4.2 单条消息体积估算（P-256 + JSON + base64）

**单个收件人小包**（aid + peer_cert_fingerprint 64位 + ephemeral_public_key 65B + nonce 12B + wrapped_message_key 48B + JSON 字段名）≈ **276 字节**

**固定头部**（type/version/aad/ciphertext/tag/sender_signature/sender_cert_fingerprint/recipients_digest 等）≈ **500 字节**

| 场景 | 消息体积（正文 ~100 字节密文） |
|---|---|
| P2P 单聊 | ~700 字节 |
| 群聊 10 人 | ~3.3 KB |
| 群聊 100 人 | **~28 KB** |
| 群聊 200 人 | **~56 KB** |

### 4.3 CPU 开销估算（桌面 P-256 单次 ECDH ~60 μs，移动端 ~300 μs）

**发送方**（每成员一次 ECDH）：

| 群规模 | 桌面 | 移动端 |
|---|---|---|
| P2P（1） | 忽略 | 忽略 |
| 100 人 | ~6 ms | ~30 ms |
| 200 人 | ~12 ms | ~60 ms |

**接收方**（无论群多大，始终只做自己那一份）：

- 每条消息 ~0.1 ms（桌面/移动相近）

### 4.4 规模上限（已定）

本方案为 Agent 社交场景设计，不追求大群可扩展性。

- **硬上限：200 人**。群规模 ≤ 200 人时启用 E2EE 加密
- **超过 200 人的群：强制降级为明文通信**，协议层拒绝 `group.send` 带 `encrypted: true` 的请求
- 业务层在群规模临近上限时应告知用户"超过 200 人后消息将不再加密"
- 更大规模的 Agent 群（公告、广播类）天然适合明文，端到端加密在此类场景下边际收益极低

## 5. 场景分析（全景覆盖）

本章逐个穷举群聊/P2P 生命周期里的关键场景，说明新方案是否能覆盖。

### 5.1 新成员加入群

**触发**：admin 调用 `group.add_member`，或用户通过邀请码加入。

**处理**：
- 服务端把新成员写入成员表，向全体在群成员推 `group.members_changed` 事件
- 现有成员客户端增量拉取新成员证书并缓存
- 新成员客户端首次对该群发消息前，批量拉一次全员证书
- 此后发送方在给该群发消息时，收件人清单自动包含新成员

**新成员看不到加入前的历史消息**：因为旧消息的收件人清单里没有他，他的 AID 私钥配不上任何一项。这是协议层天然的"加入前历史隔离"。

**能否解决**：✅ 完全解决，无需任何额外机制。

---

### 5.2 成员离群（主动退出或被踢）

**触发**：`group.leave` / `group.kick`。

**处理**：
- 服务端从成员表移除该成员，向全体在群成员推 `group.members_changed` 事件
- 其他成员客户端更新本地成员列表
- 之后的群消息，发送方组装清单时自动跳过该成员
- 该成员已下载到本地的历史消息仍可读（明文存在本地，协议层无法收回，这是任何端到端加密方案的共同现实）
- 但他**再也收不到新的群消息**（服务端不再路由给非成员；即使恶意截流，清单里也没有给他的小包）

**无需 epoch 轮换，无需重新分发群密钥**——因为根本没有"群密钥"这个概念。

**能否解决**：✅ 完全解决。

---

### 5.3 成员更换证书（证书轮换）

**触发**：Agent 主动轮换 AID 证书，或原证书被吊销后申请新证书。

**处理**：
- 新证书通过 AUN 身份服务发布
- 发送方给该成员发消息时，可能发生两种情况：
  - **发送方缓存里是旧证书**：加密后对方解不开（新私钥配不上旧公钥加密的小包）→ 对方在解密失败时懒加载触发一次证书重拉 → 但此时消息已经错过。**解决：** 发送方在组装清单前，如收到过该成员的 `cert_changed` 事件或发现 fingerprint 不匹配，拉新证书重做清单。保守策略下，服务端在 `group.send` 落地时可附加每成员当前 fingerprint 校验，不匹配则拒绝并回传新证书指纹
  - **发送方缓存里已是新证书**：直接用新公钥加密，对方正常解开

**已发出的历史消息**：用旧证书私钥加密的，旧私钥若已销毁则解不开——但由于"消息本地明文保存"这条假设，本地早已有明文，新证书后无需回解历史密文。

**能否解决**：✅ 通过懒加载 + 发送前 fingerprint 校验组合解决。

---

### 5.4 成员长期离线后再上线

**触发**：某成员手机关机一个月后重新打开。

**处理**：
- 客户端重连建立 WebSocket
- 拉取离线期间的群消息队列（服务端为每成员保留的消息，协议层现已存在）
- **对每一条未读消息独立解密**——每条消息都自包含发送时的收件人清单和对应小包，与是否在线无关
- 期间可能发生的成员变更、证书轮换——每条历史消息用的是**当时**的成员清单和证书公钥，解密只和"自己的 AID 私钥"有关，不受后续变更影响
- 上线后的第一次发送消息前，批量刷新一次成员列表和证书缓存

**对比现行方案**：现行方案离线期间若发生了 epoch 轮换，上线后需要走 `group.key_request` 找在线成员响应，常见失败；新方案完全不需要这个路径。

**能否解决**：✅ 完全解决，这是新方案相对现行方案最大的体验提升点。

---

### 5.5 成员的某个设备丢失 / 被盗

**触发**：用户手机丢失，担心别人能看群消息。

**处理**：
- 由于全设备共用同一 AID 证书私钥，丢失设备 = 该 AID 私钥可能泄露
- 用户**必须执行 AID 证书轮换**（走 §5.3 流程），重新发布新证书
- 轮换后，旧证书私钥自动失效，丢失的设备无法解密**之后**的群消息
- 丢失设备上**已有**的本地明文无法远程清除（这是设备层问题，协议无法解决；和 Signal、WhatsApp 一样）

**能否解决**：✅ 协议层通过证书轮换解决后续风险；本地明文泄露属于设备安全问题，不在协议责任范围。

---

### 5.6 服务端恶意篡改成员列表 / 注入幽灵成员

**威胁**：服务端把收件人清单里加一个攻击者 AID，企图让攻击者也能收到消息。

**处理**：
- 发送方的 `sender_signature` **必须覆盖完整的收件人清单**（议题 #4 的方向性结论）——服务端任何修改都会导致验签失败
- 接收方验签时一并校验"自己 AID 在清单里" + "签名有效"
- 因此服务端**不能新增收件人**（会破坏签名），也**不能剔除收件人**（只是让被剔除者收不到消息，但消息已在全体接收方那里留证）

**能否解决**：✅ 签名覆盖清单即可解决。

---

### 5.7 发送方与接收方的时间差（消息到达时成员已变更）

**威胁**：Alice 在 T0 时刻按当时成员清单发送消息；T1 时刻 Bob 被踢出；T2 时刻消息送达剩余成员。Bob 还能不能看到？

**处理**：
- 若服务端在 T1 之前已把消息分发给 Bob → Bob 能看到这条消息（在他被踢之前合法收到，事后无法追回）
- 若服务端在 T1 之后才处理这条消息 → 服务端在路由时可选择丢给还在群里的成员，或直接拒绝（根据业务策略）
- 这是**所有 IM 协议都有的语义时间窗**，不是 E2EE 特有问题

**能否解决**：✅ 协议层无需特殊处理，按业务语义定。

---

### 5.8 发送方同时踢人 + 发消息的竞态

**威胁**：Alice 按旧成员清单（含 Bob）构造消息，同时发出 `group.kick(Bob)`，两条 RPC 的顺序可能交错。

**处理**：
- 服务端在 `group.send` 落地时比对"清单覆盖的 AID" vs "当前成员"——如果 Bob 还在清单里但已被踢出，服务端两种策略：
  - 宽松：忽略多余的 Bob 小包，正常转发（Bob 已非成员，不会被路由）
  - 严格：拒绝 `group.send`，让客户端重做清单重试
- 推荐宽松策略：多余小包占带宽但不影响正确性

**能否解决**：✅ 服务端冲突处理即可。

---

### 5.9 两个客户端发送时读到的成员列表不一致

**威胁**：Alice 和 Carol 同时发消息，Alice 看到 10 人，Carol 看到 11 人（期间加了一人）。

**处理**：
- 新加入的那个人：Alice 的消息里没有他的小包 → 他解不开 Alice 那条消息；Carol 的消息里有 → 他能解 Carol 那条
- 这种"加入瞬间消息可见性不一致"在任何 IM 都存在，不是 E2EE 问题
- 若业务要求"一旦加入就能看到加入前瞬间所有消息"，需要在应用层做历史开放（本方案不支持，协议原则是"加入前历史隔离"）

**能否解决**：✅ 符合协议"加入前隔离"原则。

---

### 5.10 证书吊销（紧急撤销）

**触发**：某 Agent 证书被 CA 吊销（被判定为失陷）。

**处理**：
- 其他成员通过 CRL / OCSP 感知吊销
- 发送方在发消息前检查成员证书状态；已吊销的成员被排除在清单外（或等同"已离群"处理）
- 若服务端缓存了群成员的吊销状态，可在 `group.send` 落地时附加检查
- 吊销前对方下载的历史明文无法撤回（同 §5.5）

**能否解决**：✅ 通过 CRL/OCSP + 发送前检查解决。

---

### 5.11 超过 200 人上限的群

**触发**：群人数增长到 200 人以上。

**处理**：
- 协议层拒绝 `group.send` 带 `encrypted: true`
- 业务层提示用户"群规模超过 200，消息不再加密"
- 降级为明文 `group.send`
- 若群规模回落到 200 人以内（有人离开），可重新启用加密；此时由发送方自行决定，不需要全群协同

**能否解决**：✅ 硬上限下降级明文。

---

### 5.12 场景覆盖总结

| 场景 | 新方案 | 对比现行方案 |
|---|:---:|---|
| 新成员加入 | ✅ | 现行需 epoch 轮换 + 分发，新方案自然生效 |
| 成员离群 / 踢人 | ✅ | 现行需 epoch 轮换，新方案自然生效 |
| 证书轮换 | ✅ | 现行依赖懒加载，新方案同 |
| 长期离线后上线 | ✅✅ | **现行严重受影响**（key_request 失败常态），新方案无此问题 |
| 设备丢失 | ✅ | 两方案都靠证书轮换 + 本地不可回收明文 |
| 服务端篡改清单 | ✅ | 签名覆盖清单解决，现行靠 manifest 签名 |
| 时间窗竞态 | ✅ | IM 共性问题，非 E2EE 专属 |
| 发踢并发 | ✅ | 服务端冲突处理 |
| 成员列表不一致 | ✅ | 新加入者消息可见性不一致，符合预期 |
| 证书吊销 | ✅ | CRL/OCSP + 发送前检查 |
| 超大群 | ✅ | 硬降级明文 |

## 7. 进群瞬间的一次性代价

新方案把"持续的、不确定的、依赖在线成员的密钥恢复成本"换成了**"进群那一刻的一次性批量拉证书成本"**。

- 加入 100 人群：1 次批量 RPC 拉 ~500 KB 证书数据
- 加入 200 人群：1 次批量 RPC 拉 ~1 MB 证书数据

这个成本是**一次性的且可预测的**，与 HTTPS 首次访问新站点需要拉对方证书同理，可接受。

日常发送消息时零拉取（命中缓存），远优于现行方案每次发送 5 RPC。
